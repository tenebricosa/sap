Введение      

        Цель этой фазы состоит в создании концептуального проекта, который является подробной документацией по требованиям к бизнес-процессам компании. В концептуальном проекте описываются общие принципы отражения бизнес-процессов в одной или нескольких SAP-системах. Концептуальный проект подробно описывает назначение бизнес-сценариев, бизнес-процессов, шагов процессов и требования по внедрению решения SAP.

      

Управление проектом

      

1.      Управление проектом 

Цель: установить оптимальный цикл операций по управлению проектом, который бы гарантировал успешное выполнение проекта внедрения. Этот пакет определяет все операции по планированию, контроллингу и обновлению. Здесь также устанавливаются условия, при которых возникает необходимость вмешательства (по согласованию с руководством отдела) во взаимоотношения между бизнес-процессами и организационной структурой.

 

Обучение проектной группы

2.      обучение проектной группы 

Цель этого пакета работ состоит в завершении подготовки и в проведении обучения проектной группы. Также в рамках обучения на фазе концептуального проекта следует завершить разработку стратегии документации и обучения конечных пользователей, а также подготовить прототип поставляемых результатов по обучению и документации. 

Необходимо провести обучение членов проектной группы и группы по изменениям для распространения функциональных и технических знаний SAP. Обучение позволяет проектной группе приступить к согласованию требований системы с функциями SAP.

       

 Разработка системной среды

Цель данного процесса состоит в том, чтобы инсталлировать и технически сконфигурировать тестовый мандант и системы разработок. В рамках данного процесса необходимо определить и проверить процедуры системного администрирования для системы разработки.

В рамках процесса выполняются следующие операции:

·        Создание проекта технической среды

·        Настройка среды разработок

·        Начальная настройка системного ландшафта

·        Системное администрирование

·        Инициализация IMG

3.      Создание проекта технической среды

Цель этой операции состоит в определении и документации технического проекта. В этой работе рекомендуется воспользоваться помощью вашего партнера по аппаратному обеспечению.

Руководящий комитет должен подтвердить, что проект технической системной среды удовлетворяет требованиям внедрения R/3.

4.      Настройка среды разработок 

 Цель: инсталлировать и сконфигурировать техническую среду для системы разработок.

 Что необходимо сделать.

·       Установить начальные аппаратные средства (АС): воспользовавшись услугами поставщика АС, установить необходимые аппаратные средства и дополнительное вспомогательное оборудование, определенное на этапе выработки технических требований (например, принтеры, дополнительные серверы, сетевые устройства, маршрутизаторы), и присоединить все эти средства к существующей сети.

·       Инсталлировать и сконфигурировать манданты системы разработок: инсталлировать инстанции приложений и БД для системы технической изолированной среды (не обязательно) и системы разработок, как это описано в Руководстве по внедрению R/3.

 Подробно этот вопрос будет рассмотрен в главе администрирования. 

  • Создать основные записи пользователей для проектной группы: создание основных записей пользователей для членов проектной группы с соответствующими профилями полномочий, необходимыми для системы разработки. Эти профили должны быть созданы в SAP-манданте-шаблоне 001, чтобы их можно было переносить в другие манданты, когда последние будут создаваться. Не следует отстранять ни одного члена проектной группы от выполнения любых функций бизнес-приложений в настроечном манданте среды разработок. Это будет способствовать тому, что все они смогут полностью смоделировать бизнес-процессы в вашей системе без каких-либо ограничений. Сначала рекомендуется создать новый профиль (<ВашаКомпания>_ALL), смоделированный на базе профиля SAP_ALL, но без полномочий суперпользователя.

5.      Начальная настройка системного ландшафта 

 Цель данной операции состоит в том, чтобы настроить манданты системы разработок. Чтобы понять структуру мандантов SAP и роли, которые каждый мандант или система играет в процессе внедрения R/3, следует ознакомиться с соответствующей документацией SAP. При выполнении данной операции необходимо определить управление мандантами. Настройте управление мандантами, чтобы отобразить функции ваших мандантов, указав следующие параметры:

  • автоматический перенос
  • роли мандантов
  • ведение для всех мандантов

Подробно этот вопрос будет рассмотрен в главе администрирования.

 6.      Системное администрирование

 

 Цель данной операции состоит в том, чтобы определить, проверить и использовать процедуры системного администрирования для системы разработок.

 Что необходимо сделать.

Определить системное администрирование для системы разработок: определить операции и процедуры системного администрирования для системы разработок путем использования ссылочного проверочного списка, предоставляющего для каждой операции необходимые процедуры. Задокументируйте каждую операцию и присвойте ее соответствующему члену группы. Каждый такой "владелец" операции отвечает за ее выполнение и документирование.

Проверить функции системного администрирования на тестовом манданте. Каждая процедура должна быть проверена по следующим позициям:

·       Системный(ые) администратор(ы) R/3 обучен

·       Система R/3 правильно сконфигурирована

·       Описание составлено достаточно подробно, четко и полно

·       Полученный результат совпадает с ожидаемым

·       Выполнение возможно в различных обстоятельствах

 7.      Инициализация IMG

 

 Цель данной операции состоит в создании Руководства по внедрению (IMG). Важно создать IMG таким образом, чтобы максимально облегчить процесс настройки и документирования. IMG используется в качестве ссылки на последних этапах определения бизнес-процессов.

В управлении проектами имеется два способа генерации IMG проекта:

·        Путем выбора атрибутов

·        Путем выбора вручную тех компонентов, которые вы хотели бы включить в ваш ракурс  IMG.

Определение организационной структуры предприятия

8.      Определение организационной структуры предприятия

 

Цель данной операции состоит в подготовке семинаров по организационной структуре предприятия.

 Назначение операции состоит в том, чтобы определить организационную структуру предприятия, используя организационные единицы, принятые в SAP, такие как балансовые единицы и сбытовые организации.

На семинаре по обсуждению организационной структуры обсуждается несколько вопросов, в том числе:

·        Какие сферы деятельности предприятия и бизнес-процессы будут использованы в системе

·        Какие организационные единицы в рамках компании будут охвачены внедрением

Данный процесс  поддерживается главным образом базой данных вопросов и ответов (БД В&О).

Определение бизнес-процессов

В рамках процесса выполняются следующие операции:

  • Подготовка к семинарам по бизнес-процессам

  • Проведение семинара по глобальным требованиям

  • Проведение семинаров по бизнес-процессам

  • Завершение формирования концептуального бизнес-проекта

 

 

После определения организационной структуры, вы определяете бизнес-процессы компании в терминах системы R/3.

 База данных вопросов и ответов представляет собой инструмент, поддерживающий определение бизнес-процессов. Для поддержки подготовки и завершения определения бизнес-процесса вы можете использовать форму описания исходных данных клиента и вопросы по бизнес-процессам в Базе данных вопросов и ответов. С целью подробного анализа можно привлекать посредством БД вопросов и ответов внешние инструменты моделирования бизнес-процессов.

Генерация концептуального бизнес-проекта объединяет в едином документе все требования к бизнес-процессам компании. Здесь же создается документ «Основной список бизнес процессов» для фазы реализации.

9.      Подготовка к семинарам по бизнес-процессам

 Цель данной операции состоит в том, чтобы обеспечить эффективность семинаров по бизнес-процессам. Это означает, что необходимо определить, кто должен присутствовать, какие бизнес-процессы и связанные с ними вопросы будут обсуждаться. Чтобы заседания были эффективны, необходимо заранее разослать всем участникам необходимые материалы и инструкции по бизнес-процессам.

10.      Проведение семинара по глобальным требованиям

 

 Цель данной операции состоит в том, чтобы проанализировать цели стандартизации (глобальные параметры).

 Поддерживаемые глобальные параметры определяют настройки, которые редко придется изменять и которые должны соответствовать стандартам ISO. Обмен данными с внешними системами особенно чувствителен к изменениям в согласованных международных настройках.

 В числе примеров глобальных параметров, нуждающихся в стандартизации, можно назвать настройки для страны, календарь, валюты и т.д.

В число других стандартизируемых параметров входят: системы нумерации, планы счетов, основные данные и т.д.

11.      Проведение семинаров по бизнес-процессам

 

 Цель данной операции состоит в том, чтобы провести семинары по бизнес-процессам для определения всех требований, включая: бизнес-процессы, систему отчетов, интерфейсы, преобразования, расширения, полномочия.

Все полученные, в результате проведения семинаров, материалы войдут затем в концептуальный бизнес-проект. Для определения бизнес-требований и заполнения формы описания исходных данных клиента необходимо использовать вопросы из Базы данных вопросов и ответов, которые можно найти, в зависимости от сферы деятельности, в соответствующей ветви дерева «Ссылочная структура SAP». 

Вопросы, касающиеся организационной структуры, зависят от точки зрения на содержимое соответствующей сферы бизнес-процессов.  

Вопросы, касающиеся конкретных бизнес-процессов, также зависят от точки зрения на содержимое соотвествующего бизнес-процесса. Ответ может быть либо сформулирован произвольно, либо дан в виде:

·        да, нет, затрудняюсь ответить

·        Список ответов

 Вопросы, касающиеся формы описания исходных данных клиента, одинаковы для всех бизнес-процессов, и на них можно отвечать в произвольной форме.

На каждом уровне структуры может быть назначено ответственное лицо.

 Описание бизнес-процессов завершается включением транзакций соответствующих бизнес-процессов в объем проекта внедрения. Число транзакций, соотнесенных с данным бизнес-процессом, отображается в матрице транзакций бизнес-процесса, когда вы выбираете в меню Транзакции бизнес-процесса.

В эту матрицу входят:

·        Код: имя транзакции бизнес-процесса

·        Транзакция: подробный текст транзакции бизнес-процесса

·        Имя транзакции бизнес-процесса

·        Тип транзакции бизнес-процесса (TR- транзакция, DA- основные данные, RP- отчет)

·        Объем: показывает, что транзакция бизнес-процесса входит в объем

·        Прототип: указывает транзакцию бизнес-процесса в рамках объема прототипа.
Прототип определяет базовый объем транзакций, которые будут внедряться.

·        Z1-Z4 описывают предлагаемые циклы кофигурационных тестов.

·        I1-I2 описывают предлагаемые циклы интеграционных тестов.

 Матрица транзакций бизнес-процесса содержит значения по умолчанию для бизнес-процесса, предлагаемые SAP. Не следует вносить никаких изменений в матрицу транзакций бизнес-процессов. Их следует осуществлять в Основном списке бизнес процессов(ОСПБ), который представляет собой файл MS EXCEL, базирующийся на матрице транзакций бизнес-процессов. Любые изменения должны делаться только при участии консультантов.

 12.      Завершение формирования концептуального бизнес-проекта

 

 Цель данной операции состоит в завершении формирования концептуального бизнес-проекта (КПБ), который является основным документом, предоставляемым в рамках проекта ASAP на фазе 2. КБП служит концептуальным основным планом и превращается в подробный письменный документ. Этот документ содержит подробно изложенные бизнес-требования и является базисом для организации, конфигурирования и, в случае необходимости, дополнительных разработок.

Проверка качества

13.      Проверка качества на фазе концептуального бизнес-проекта

 

 Цель данного пакета работ состоит в окончательной проверке всех документов, предоставляемых в рамках проекта ASAP на данной фазе. Важно, чтобы проверки качества выполнялись на протяжении всей работы по мере решения соответствующих задач. Благодаря концептуальныому бизнес-проекту обеспечивается точное понимание всеми участниками окончательного объема проекта в части бизнес-процессов, организационной структуры и системной среды. Кроме того, он включает обучение проектной группы, стандарты проекта и стратегии в технической сфере. Необходимо учитывать аспекты, относящиеся к изменениям объема, их влиянию на бюджет и планирование ресурсов.
