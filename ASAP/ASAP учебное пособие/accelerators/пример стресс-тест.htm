Пример плана проведения стресс-теста

Цель

Цель этого документа – предоставить пример плана проведения стресс-теста в продуктивной среде SAP.

Стресс-тест выполняется, чтобы подтвердить, что сконфигурированная продуктивная среда пригодна для всех бизнес-процессов при продуктивной эксплуатации, а также позволяет определить потенциальные улучшения, которые надо внести перед переходом к продуктивной эксплуатации.

Методология

· Для обеспечения точности результатов выполнение стресс-тестов проводится на реальном продуктивно работающем аппаратном обеспечении.

· Соответствующий объем и нагрузка транзакций создаются с помощью инструментария тестирования и оперативного ввода пользователями.

· Каждая итерация стресс-теста называется циклом.

· Нагрузка будет подаваться на продуктивную систему постепенно.

· Процедура управления изменениями должна тщательно контролироваться.

· Действия всех групп, участвующих в проведении стресс-тестов, должны быть скоординированы с выполнением общей работы.

Шаги

  1. Определение набора критических бизнес-процессов, для которых необходимо выполнить стресс-тест:
    1. запись заказа клиента;
    2. создание фактуры;
    3. создание поставок;

и т.д.

 

2. Определение основных технических областей, тестирование/мониторинг которых нужно выполнить:

  • производительность SAP R/3 (буферы, ответственные транзакции, время реакции, ошибки);
  • производительность уровня ОС (R/3, аппаратное обеспечение, сети, производительность системы);
  • база данных (кэш-буферы, память);
  • интерфейсы SAP;
  • печать SAP;
  • сеть.

4. Выполнение действий, предшествующих стресс-тесту:

  • определение критических бизнес областей (критических транзакций, фоновых заданий, заданий печати и интерфейсов);
  • настройка транзакций перед выполнением стресс-теста;
  • тренировка выполнения действий стресс-теста (мониторинг, настройка производительности, создание отчетов о проблемах);
  • определение стандартов мониторинга:
    • инструменты мониторинга,
    • методы сбора статистических данных,
    • инструменты для отслеживания проблем,
    • методы координации/коммуникации с другими группами;
  • установление ролей и сфер ответственности.

 

Процедуры стресс-теста

Внимание: прежде чем приступать к выполнению стресс-теста, выполните резервное копирование, чтобы после завершения теста выполнить восстановление.

 

1. Создание эталонных статистических данных

· Необходимо записывать статистические данные, чтобы сравнивать предыдущие циклы тестирования с последующими циклами.

 

2. Выполнение стресс-тестов

· Статистические данные должны аккуратно собираться и записываться, чтобы можно было провести точное их сравнение по каждому из циклов.

· Все группы должны взаимодействовать друг с другом, чтобы обеспечить согласованность и полноту выполнения тестов. К основным участникам тестирования относятся: группа приложений, техническая группа, группа разработок и сетевая группа.

 

3. Анализ результатов

· Результаты по каждой из областей мониторинга должны быть записаны и сравнены. Эти результаты также включают впечатления конечных пользователей, технические статистические данные, статистические данные в области приложений и т.д.

· На этом этапе после завершения каждого из циклов тестирования проводится совещание для сравнения и анализа результатов.

 

4. Выполнение настроек среды SAP, если необходимо

· Для эффективного тестирования необходимо, чтобы изменения строго контролировались и записывались.

· Решения о выполнении изменений принимает группа, проводящая стресс-тесты.

 

5. При необходимости повторите процедуру.

 

Стандарты мониторинга

Интервалы для сбора статистических данных – 15 минут.
  • SAP R/3: транзакции в контролируемые буферы, системный журнал, сеть.

Транзакции: ST02, SM21, SM51, SRZL.

  • Инструментарий уровня ОС для мониторинга аппаратного обеспечения, производительности R/3, сети.

Транзакции: ST06, ST03.
  • Инструментарий базы данных для мониторинга ее производительности.

Транзакции: ST04.
