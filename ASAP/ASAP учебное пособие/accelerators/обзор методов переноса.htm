Обзор технологии

Для выбора метода переноса данных из прежней системы в систему R/3 существует три возможности. Возможные технологические варианты таковы:

  • прямой ввод;
  • вызов транзакции;
  • пакетный ввод.

Выбор метода

Рекомендации по выбору метода переноса данных сгруппированы последовательно, начиная с наиболее предпочтительного метода.

Метод прямого ввода

Этот метод переноса данных из прежней системы в систему R/3 является наиболее предпочтительным. Он также является самым быстрым. В качестве исходного файла для обработки специальных прикладных функциональных модулей предоставляется последовательный файл, содержащий все данные в символьном формате. Эти функции выполняют все обычные проверки на целостность данных. После того, как записи успешно обработаны, они заносятся непосредственно в соответствующие таблицы баз данных приложений. В случае возникновения логических ошибок записи данных с ошибками передаются в процедуру обработки особых ситуаций.

Этот метод позволяет избежать проблем, связанных с диалоговой обработкой и обработкой обновления. Для всех запросов к базе данных используется методика управления массовыми записями данных.

Метод прямого ввода рекомендуется в случае необходимости любого переноса данных, особенно для транзакций с большими объемами данных.

Примечание: этот метод должен применяться во всех ситуациях, для которых существуют основанные на нем прикладные функции.

Для получения дополнительной информации по подобным прикладным функциям см. раздел Стандартные программы переноса данных R/3 .

Метод вызова транзакции

Метод вызова транзакции - это следующий по скорости метод переноса данных из прежней системы в систему R/3. В данном случае программа переноса данных обрабатывает информацию из последовательного файла и вызывает требуемую транзакцию с помощью ABAP-оператора.

Данные исходного последовательного файла обрабатываются посредством экранов приложения отдельной транзакции. Все проверки и контроль данных осуществляются с точки зрения логики приложения. Они представляют собой синхронную диалоговую обработку.

Если диалоговая обработка завершается успешно, соответствующее приложение активирует обработку обновления для выполнения всех запросов к базе данных. Обработка обновления базы данных может быть синхронной или асинхронной.

Данная технология не предоставляет возможностей обработки ошибок или создания протокола обработки транзакций. Включение процедур обработки особых ситуаций и необходимых протокольных функций, как часть разработки программы переноса методом вызова транзакции, лежит в сфере ответственности соответствующего прикладного программиста.

Метод вызова транзакции рекомендуется использовать во всех случаях, для которых не существует программ прямого ввода данных. Традиционная технология пакетного ввода может быть рекомендована только для случая дополнительной обработки ситуаций, связанных с ошибками

Метод пакетного ввода

Технология пакетного ввода является традиционным методом внедрения программ переноса данных. Преимущество этого метода перед методом вызова транзакции заключается в том, что технология пакетного ввода имеет функцию, отвечающую за администрирование и управление пакетным вводом.

Кроме того, в ней существуют утилиты мониторинга и ведения задач, связанных с этим методом. То есть дополнительных затрат на программирование для анализа особых ситуаций и задач протоколирования не требуется.

Данные последовательного файла обрабатываются посредством приложения и сохраняются в сеансе пакетного ввода с помощью функциональных модулей пакетного ввода. По определению, сеанс - это массив данных одной или нескольких транзакций. Сеансы пакетного ввода физически сохраняются в системе в массиве данных очереди пакетного ввода. Они могут содержать корректные или ошибочные записи данных.

Технология пакетного ввода включает асинхронную диалоговую обработку и синхронную обработку обновления. В отличие от метода вызова транзакции, метод пакетного ввода позволяет переносить из прежней системы в систему R/3 данные для транзакций нескольких приложений.

Однако при обработке сеансов пакетного ввода новая транзакция не может быть запущена, пока предыдущая транзакция не внесена в соответствующую базу данных приложения. Кроме того, сеансы пакетного ввода не могут создаваться параллельно.

Функции мониторинга и администрирования технологии пакетного ввода вызываются с помощью пути по меню Утилиты ® Пакетный ввод ® Обработать или с помощью транзакции SM35. Возможны три способа обработки сеансов пакетного ввода:

  • диалоговая обработка;
  • просмотр только ошибок;
  • фоновая обработка.

Программа переноса данных, использующая технологию пакетного ввода должна :

1. открывать сеанс пакетного ввода в очереди перед началом переноса в него данных;

2. закрывать сеанс пакетного ввода после перенесения в него данных.

Приведенные выше функциональные требования могут быть запрограммированы с использованием специальных функциональных модулей, предоставляемых технологией пакетного ввода.

Этот метод предлагает широкие возможности управления ошибками, а также подробного анализа протокола транзакций.

Оценка

Каждый раз необходимо проводить определенное тестирование процесса переноса данных и оценивать, какой из методов является наиболее адекватным, принимая во внимание объем данных, размер и производительность системы, а также доступные ресурсы.

При прочих равных условиях, технология прямого ввода должна использоваться во всех случаях, когда соответствующие функции доступны, или же необходимо разработать новые функции переноса данных.

Выбирая между технологиями вызова транзакции и пакетного ввода, вызов транзакции всегда следует использовать в первую очередь, и лишь затем обрабатывать все особые случаи с помощью методики пакетного ввода. Такая стратегия настоятельно рекомендуется при внедрении любых новых программных решений переноса данных из прежних систем.

Выводы

Метод прямого ввода

Этот метод переноса данных следует применять во всех случаях.

Метод вызова транзакции или метод пакетного ввода

Традиционно существует некоторая неопределенность в выборе между методами пакетного ввода и вызова транзакции. Всегда следует анализировать оба метода, используя разработанные в них функций тестирования. На процесс такого выбора оказывают влияние многие параметры, учитывающие конфигурацию конкретной системы. В качестве общего правила можно предложить следующее:

В тех случаях, когда в рамках проекта недостаточно времени на внедрение фазы переноса, включающей большие объемы данных, для программирования и обработки информации из прежних систем следует использовать метод вызова транзакции.

С другой стороны, если для реализации проектов имеется достаточное время, то для внедрения и обработки решений по переносу больших объемов информации из прежних систем в систему R/3 используется метод пакетного ввода.

Следует заметить, что во всех случаях, в особенности для метода вызова транзакции, предпочтительным является разработка решения, использующего оба метода в сочетании.

Данные из прежней системы, подлежащие переносу в систему R/3, тестируются и проверяются. Если данные не содержат ошибок, то для их обработки следует применять метод вызова транзакции и вызывать метод пакетного ввода только для обработки особых случаев.

В заключение, всегда необходимо анализировать оба возможных метода - вызов транзакции и пакетный ввод - и соответствующим образом внедрять выбранное решение с учетом ограничений конкретной среды внедрения.
