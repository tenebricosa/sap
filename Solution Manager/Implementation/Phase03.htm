Введение      

        Цель фазы реализации состоит в конфигурировании и тестировании внедренного решения. Задачами являются окончательная конфигурация системы, полное тестирование и деблокирование системы для продуктивной эксплуатации.

        На этой фазе осуществляется определение и тестирование параметров настройки с использованием основных данных и организационных единиц, включенных в объем на фазе концептуального проекта.

        На фазе реализации проекта в SAP Solution Manager должны быть выполнены следующие операции (см. рис. 2.15):

Конфигурирование бизнес-процессов

        С использованием транзакции SOLAR01 можно получить доступ к различным инструментальным средствам настройки. Это центральная точка доступа для конфигурирования системных настроек в рамках проекта внедрения.

        Используемые в структурах объекты из Репозитария бизнес-процессов могут быть уже присвоены транзакциям и пакетам настройки бизнес-конфигурации. При создании структур без использования объектов из Репозитария бизнес-процессов возможно самостоятельное присвоение транзакций и пакетов настройки бизнес-конфигурации, объектов IMG и контрольных примеров.

        Конфигурация включает следующие шаги (см. рис. 2.16).

 

 

 

 

 

Рисунок 2.16 – Конфигурация

    Присвоение транзакций/программ

        Возможен вызов присвоенных в концептуальном проекте транзакций, добавление новых или удаление лишних транзакций из объема проекта.

  Присвоение/обработка пакетов настройки бизнес-конфигурации

        Имеется возможность присвоения этапам проекта пакетов настройки бизнес-конфигурации, просмотра  содержимого присвоенных пакетов и удаление из объема ненужных пакетов. Также можно активировать один или все пакеты настройки бизнес-конфигурации, присвоенные определенному процессу. В прикладной системе возможно создание и присвоение в SAP Solution Manager новых пакетов настройки бизнес-конфигурации для проекта или специфичных для компании параметров настройки, например, в целях развертывания в рамках концерна или для создания собственных решений.

 Присвоение IMG

        Этапам проекта могут быть присвоены объекты IMG. Эти параметры настройки, а также параметры настройки, созданные при помощи пакетов настройки бизнес-конфигурации в руководстве по внедрению, можно просматривать и обрабатывать.

Обработка контрольных примеров

        После конфигурации возможно проведение начального тестирования функций. Имеется возможность присвоения контрольных примеров, сценариев, процессов или шагов процессов для последующего использования в ориентированных на процессы тестах. Возможно присвоение существующих контрольных примеров или создание новых. Для создания, изменения и просмотра доступны следующие виды контрольных примеров:

  • Описание контрольного примера в форме документа Microsoft Word;

  • Описание контрольного примера в форме веб-ссылки (URL);

  • Контрольные примеры, выполняемые вручную;

  • Внешние приложения, например, внешние инструментальные средства тестирования с сертифицированным интерфейсом или документы MicrosoftWord;

  •  Автоматические контрольные примеры, например, конфигурации тестов CATT или eCATT;

  • Тесты функциональных модулей.

Распределение настройки. Проверка непротиворечивости      

        В системной среде могут присутствовать несколько компонентов SAP, например, системы R/3 и SAP CRM. В основе операций в системе SAP CRM лежат данные, уже имеющиеся в системе R/3; эти системы взаимодействуют во многих бизнес-сценариях. Параметры пользовательской настройки, определяющие бизнес-процессы в системах, должны быть синхронизированы для обеспечения корректного и бесперебойного обмена данными. Области, в которых должны быть синхронизированы настройки систем R/3 и SAP CRM, включают клиента и группы продуктов, рынки сбыта, условия платежа, условия поставки, группы комиссионных, валюты,страны, регионы и единицы измерения.

        После внедрения и конфигурирования системной среды были также настроены различные прикладные компоненты SAP. Системная среда включает различные компоненты, имеющие собственные независимые таблицы настройки (с технической точки зрения). Для этих компонентов по практическим соображениям также требуется совместная настройка.  В идеальном случае система CRM использует данные системы SAP R/3 Enterprise. В определенных областях конфигурация этих двух систем должна совпадать. Совместная настройка должна быть синхронизирована. В частности, изменения в системе R/3 могут потребовать изменений в прикладном компоненте SAP.

        Для синхронизации выбранных параметров пользовательской настройки в исходной системе (например, системе R/3) с параметрами пользовательской настройки в целевых системах (например, системе SAP CRM, APO и т.п.) можно воспользоваться функцией распределение настройки (Customizing distribution).

        Распределение настройки инициирует процесс переноса параметров настройки в целевые системы.Для переноса изменений настройки из систем разработок в системы обеспечения качества (QAS) и продуктивные (PRD) системы в рамках функции распределения настройки используются запросы на перенос. Распределение настройки выполняется только между системами разработок (DEV).  Принцип работы функции распределения настройки демонстрирует рисунок 2.17.

Рисунок 2.17 – Принцип работы распределения настройки.

        Распределение настройки позволяет выполнять в отношении параметров пользовательской настройки в среде разработок следующие функции:

  • Начальная выгрузка из исходной системы в целевую систему (начальное распределение);

  • Запланированное распределение;

  • Автоматическое распределение после каждого деблокирования переноса или изменения настройки (из системы R/3 4.6C);

  • Распределение вручную.

        Процедура распределения настройки используется для синхронизации пользовательской настройки при внедрении или обновлении системной среды. Прежде чем синхронизировать пользовательскую настройку, можно выполнить сравнение параметров настройки с помощью другого инструментального

        В системе предварительно необходимо определить конкретные объекты, по которым должна быть выполнена синхронизация – объекты настройки (синхронизации). Редактор групп синхронизации позволяет определить группы объектов группы синхронизации, для которых должна быть выполнена корректировка путем распределения настройки или сравнение с помощью инструмента Customizing Scout. В редакторе доступны такие функции, как создание, обработка, копирование, удаление и перенос.

        Объекты синхронизации могут представлять собой объекты настройки следующих типов:

  • Таблица (табличная область);

  • Ракурс;

  • Табличная область с таблицей текстов;

  • Кластер ракурсов;

  • Транзакции;

  • Логические объекты переноса.

        Порядок распределения настройки отражает рисунок 2.18.

Рисунок 2.18 – Распределение настройки

  1. Изменение параметров;

  2. Запись изменений в запрос на перенос;

  3. Сохранение изменений в базе данных;

  4. Уведомление центральной системы об изменении параметров;

  5. Чтение измененных исходных параметров настройки;

  6. Создание целевых параметров настройки;

  7. Рассылка целевых параметров;

  8. Активация настройки параметров в целевой системе;

  9. Передача журнала о настройке в центральную систему.

        После распределения настройки необходимо проверить, было ли распределение выполнено успешно. Журналы распределения настройки предоставляют обзор успешно выполненных и не выполненных распределений настройки. Эта функция позволяет выяснить причины ошибок в неуспешно выполненных распределениях настройки и повторно запустить распределение настройки вручную. Имеется возможность проверки всех журналов распределения настройки и отслеживания операций по этим журналам.

        Необходимо регулярно выполнять проверку непротиворечивости настройки с целью обеспечения синхронизации настройки системы R/3 с настройкой прикладного компонента. SAP Solution Manager включает в себя инструмент Customizing Scout, позволяющий сравнить различные параметры настройки в SAP. Объектами сравнения для Customizing Scout являются те же определенные заранее объекты синхронизации.

 Организация тестирования

        Операции по организации тестирования проводятся после создания концептуального проекта и начальной конфигурации в фазе реализации. Основные работы по тестированию изображены на рисунке 2.19.    

Рисунок 2.19 – Организация тестирования.

        Первой задачей при организации тестирования является создание плана тестирования. В фазе концептуального проекта была создана структура проекта, содержащая бизнес-сценарии, процессы и задачи процессов. После этого задачам процессов были присвоены транзакции и контрольные примеры. При создании плана тестирования система предлагает эту структуру проекта в качестве основы для создания планов тестирования. Также предоставляются все контрольные примеры (выполняемые как вручную, так и автоматически), присвоенные процессам или задачам процессов. Эти примеры можно выбрать для включения в план тестирования.

        Управление планами тестирования предоставляет следующие возможности:

  • Создание, изменение, копирование или удаление планов тестирования;

  • Перенос планов тестирования;

  • Разделение планов тестирования на пакеты тестов и присвоение этих пакетов тестерам;

  • Контроль динамики процесса при помощи инфо-системы статусов и экспорт результатов тестирования.

        Планы тестирования можно изменить, поместив новые контрольные примеры, которые еще не были помещены в структуру проекта, в существующие планы тестирования, или путем реорганизации контрольных примеров. Изменения структуры плана тестирования не переносятся автоматически на структуру проекта.

        Контрольные примеры для каждого тестера группируются в пакеты тестов, который затем присваивается тестеру. Контрольные примеры для пакета тестов выбираются из плана тестирования. Таким образом, после вызова тестером инструментальных средств тестирования на экран выводится обзор контрольных примеров, присвоенных этому тестеру в его рабочем списке (см. рис. 2.14).

Рисунок 2.20 – Пакеты тестов.

        Возможны многократные присвоения пакетов тестерам и наоборот. Для выполнения одного контрольного примера дважды независимо друг от друга необходимо скопировать пакет тестов и присвоить каждый пакет разным тестерам.

        Доступны  административные функции для:

  • Проверки рабочих списков тестеров;

  • Получения информации по загрузке и выгрузке;

  • Просмотра данных проекта для проекта с присвоенным планом тестирования;

  • Проверки системной среды проекта, которой присвоен план тестирования.

        Существует возможность вывести отчет по динамике тестирования в проекте по процессам для всех планов тестирования. Он позволяет выяснить, в каком плане тестирования и пакете тестов используется данный контрольный пример (если используется вообще).

        Анализ тестирования с помощью инфо-системы статусов позволяет контролировать динамику процесса одного или нескольких планов тестирования. Статус плана тестирования, как правило, является результатом выполнения отдельного теста. Анализы статуса, не зависящие от плана тестирования, сочетают результаты различных планов тестирования и могут считаться общим результатом комплексного тестирования, включающего несколько планов тестирования.

        Итогом является актуальный анализ статуса всех пакетов тестов в выбранном плане тестирования. К передаваемому анализу статуса можно добавить комментарий (с помощью офисной почтовой программы).

        В SAP Solution Manager имеется система сообщений о проблемах, в которой тестеры могут создавать сообщения о проблемах. Сообщения о проблемах создаются тестерами в случае невозможности выполнения теста.

        В обзоре проблем содержится следующая информация:

  • Статус сообщения о проблеме;

  • Приоритет сообщения о проблеме;

  • Автор сообщения;

  • Текущий обработчик;

  • Краткий текст сообщения;

  • Связанный контрольный пример и план тестирования.

        В обзоре проблем можно обработать список всех сообщений о проблемах, относящихся к определенному плану тестирования. Доступны следующие функции:

  • Просмотр;

  • Печать;

  • Фильтрация;

  • Экспорт в приложения Microsoft Office;

  • Сохранение в локальный файл;

  • Передача по электронной почте.

 Анализ проекта

        Существует возможность сохранения информации о проекте по фазам или операциям проекта, например, по статусу, по календарному планированию или по членам проектной группы. На этой информации основан анализ проекта. Чем больше подробной информации записано, тем большее количество опций анализа проекта имеется в наличии. Можно анализировать проект, ракурс проекта (только для проектов настройки) или нескольких проектов одновременно, в зависимости от выбранного вида проекта.

Анализы проекта включают следующие программы.

Концептуальный проект

  • Администрирование: анализ информации по администрированию и рабочих списков, которые могут быть присвоены одному или нескольким членам проектной группы. При анализе по рабочим спискам основное внимание уделяется определенным данным, например, статусу и плановому сроку окончания.

  • Присвоения: анализ присвоений в структуре проекта, которые могут являться документацией (общей или проектной) и поставляемыми или добавленными в течение проекта транзакциями.

Конфигурация

        Анализ аналогичен таковому на фазе концептуального проекта, где также можно анализировать объекты конфигурации (например, пакеты настройки бизнес-конфигурации, САТТ, IMG-операции), контрольные примеры и сообщения о проблемах с присвоениями. Кроме этого, возможен общесистемный анализ IMG проекта в системах компонентов на локальном уровне.

  • Организация тестирования: анализ планов тестирования (по отдельности или совместно)

Служба поддержки

        Можно выполнить отчет по созданным сообщениям о проблемах и просмотреть статус этих сообщений. Эта задача может включать в себя, например, следующие сценарии использования:

  • Анализ сообщений о проблемах со статусом "Открыто";

  • Анализ сообщений о проблемах, созданных определенным пользователем;

  • Проверка статуса обработки всех сообщений с приоритетом "Очень высокий".

        Анализ проекта и опции просмотра результата анализа обладают гибкой конфигурацией. Для ограничения объема анализа можно установить критерии выбора. Кроме того, можно определить способ вывода результатов анализа. Можно создать стандартные анализы и сохранить их как варианты выбора. Возможен вызов выведенных элементов структуры проекта или IMG-операций непосредственно в результатах анализа.

        В зависимости от указанных критериев выбора, при анализе проектов доступны следующие виды анализа:

  • Информация по проекту: информация по статусу и анализ видов документации;

  • Контроль динамики, например, путем создания различных комбинаций информации по статусу (например, члены проектной группы и определенный статус) или путем агрегирования информации по проекту;

  • Общие анализы, например, ежедневные анализы;

  • Отчеты по администрированию используются для отслеживания динамики различных фаз проекта и экспорта в Microsoft Excel/Microsoft Project;

  • Отчеты по присвоениям используются специально для документов и сообщений с целью получения их обзора;

  • Доступные варианты отчета необходимо адаптировать в соответствии с требованиями проекта.

        Анализ проекта можно сделать по всем проектам и фазам или для конкретного проекта или определенной фазы.
