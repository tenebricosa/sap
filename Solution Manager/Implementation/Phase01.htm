Введение      

        Целью этой фазы является обеспечение первоначального планирования и подготовки проекта SAP.  В транзакции управления проектом SAP Solution Manager выполняются все центральные административные задачи:

  • Создание проекта;

  • Определение общих данных;

  • Определение объема проекта;

  • Определение данных запросов на перенос (в зависимости от вида проекта) (см. рис. 2.2).

Проект внедрения

        Целью проекта внедрения является внедрение решения в одном местоположении. При создании проекта внедрения можно выбрать предварительно определенные сценарии из шаблонных проектов.

Шаблонный проект

      Целью шаблонного проекта является внедрение решения в нескольких местоположениях. При создании шаблонного проекта можно выбрать предварительно определенные сценарии. Шаблонные проекты можно переносить в другие системы SAP Solution Manager.

Проект оптимизации

        Проект, разработанный с целью оптимизации потока бизнес-процессов или методов применения программных решений.

Проект защиты

        Проект, разработанный с целью разрешения критических ситуаций, возникающих в ходе внедрения или использования решения. Проекты защиты позволяют выявить причины возникновения критической ситуации и скоординировать меры, необходимые для разрешения проблемы.

Проект апгрейда (обновления)

        Позволяет подготовить, структурировать и спланировать обновления.

        Выбранный для проекта вид непосредственно определяет варианты обработки, доступные на различных фазах проекта (концептуальный проект, реализация).

        При создании проекта в SAP Solution Manager можно ввести следующие общие данные.

        Руководитель проекта (Project Leader). В администрации проекта системы SAP должны существовать ответственные сотрудники. Для изменения этих присвоений необходимо воспользоваться справкой по допустимым значениям ввода, т.е. все сотрудники должны быть зарегистрированными администратором пользователями SAP. Здесь можно ввести одновременно ответственных со стороны заказчика и консультантов.

        Язык проекта (Project Language). Выбранный язык проекта используется для подготовки и вывода всей зависящей от языка информации (проектная документация и информация по статусу) для всех сотрудников проекта. Этот язык применяется независимо от языка входа в систему того или иного сотрудника проекта. Структура проекта создается на определенном языке проекта и все содержимое хранится на выбранном языке проекта. После выбора и сохранения язык проекта изменить нельзя.

        Описание (Project Description). Можно создать новый документ с описанием проекта или вставить ссылку на существующий документ.

       Статус (Status) проекта. Статус проекта позволяет контролировать состояние проекта, его завершенность. Только что созданному проекту присваивается статус "Открытый".

        Плановые данные (Plan Data). Плановые сроки начала и окончания проекта, а также объем работ в человеко-днях (MD).

        Фактические данные (Actual Data). Фактические сроки начала и конца проекта и объем фактических работ в человеко-днях (MD).

Плановые и фактические данные относятся ко всему жизненному циклу проекта.

Определение объема проекта

       Структура проекта определяется путем указания объема проекта, в который входят только те элементы, которые относятся к выбранному шаблону или шаблонам. Объем проекта можно установить различными способами в зависимости от выбранного вида проекта.

            Для проекта внедрения объем можно установить путем выбора шаблонов SAP, пользователя и/или партнера, созданных в шаблонном проекте, либо создать структуру проекта при помощи элементов, содержащихся в Репозитарии бизнес-процессов или концептуальном проекте.

            Для шаблонного проекта – путем выбора одного или нескольких решений SAP и/или партнера в качестве основы проекта. Либо без выбора существующих шаблонов. Тогда решение должно быть полностью определено в документе концептуального проекта. Это решение может быть централизованно отражено в запросах на перенос и затем распределено по другим системам.

            На этом же этапе следует определить маршрутную карту проекта. В зависимости от объема проекта, можно выбирать различные маршрутные карты для различных решений или прикладных компонентов SAP. Список операций остается неизменным, но акселераторы, связанные с этими операциями, будут применяться к конкретному решению или прикладному компоненту SAP. Каждому проекту можно присвоить только одну маршрутную карту.

           В SAP Solution Manager существует возможность контроля динамики проекта по определенным в маршрутной карте и проекте вехам (критическим точкам): после выбора маршрутной карты для проекта, на одной из закладок SAP Solution Manager появится список этапов выбранной маршрутной карты.

Назначение членов проектной команды

Каждому проекту может быть присвоено любое количество членов проектной команды. Это присвоение выполняется руководителем проекта в транзакции управления проектом. Для присвоения пользователей можно выбирать из списка пользователей системы SAP, или вводить вручную. С помощью специальной функции можно проверить, являются ли перечисленные на этой закладке члены проекта пользователями системы SAP.

        Рядом с каждым пользователем можно ввести описание сотрудника проекта. Это поле доступно только для внутреннего использования в рамках проекта. Например, оно может использоваться для документального подтверждения присвоения члена проектной команды подпроекту или проектной группе. При этом, если пользователь определен в системе SAP, имя пользователя подставляется автоматически.

        Также можно определить имена и подробную информацию о партнерах SAP, вовлеченных в проект.

         Ключевые слова. Для проекта могут быть определены ключевые слова, по которым позже может быть выполнен поиск в транзакции Анализ проекта . Ключевые слова должны быть краткими и точными.

        Виды документации.  Для всех видов проектов доступны различные форматы представления документации. Обзор форматов представления документации представлен ниже.

        Для каждого вида документации возможно создание шаблонов проектной документации на закладке. Эти шаблоны позволяют определить внешний вид и формат документации и используются для создания новых документов на фазах концептуального проекта и реализации.

        При необходимости с помощью атрибутов проектной документации можно определить возможность использовать данный вид документации в других проектах; получение возможности многократного присвоения вида документации элементам в структуре проекта (это позволяет добавлять, например, несколько документов Microsoft Word к одному структурному элементу).

        Как уже отмечалось ранее, шаблоны документации хранятся в локальной системе Knowledge Warehouse в SAP Solution Manager, а не в таблицах SAP Script.       

         Значения статуса, ключевых слов и видов проектной документации задаются по умолчанию в транзакции управления проектом. Если проекту не присвоены эти значения, администратор может использовать шаблон, содержащий не зависящие от проекта значения статуса, ключевые слова и виды документации. Значения в шаблоне проекта подходят ко всем проектам. Решение о том, какие именно значения из шаблона следует использовать и присвоить проекту, принимается администратором проекта.       

 Определение запросов на перенос проекта

        В SAP Solution Manager предусмотрены функциональные возможности для переноса шаблонных проектов. В транзакции управления проектом можно проверить запросы на перенос (подробнее о запросах на перенос см. курс лекций "Платформа SAP NetWeaver"), относящиеся к данному проекту. Также для шаблонных проектов для переноса определяется пакет. Структура проекта и присвоенные ему элементы (бизнес-содержимое) присваиваются указанному пакету. После этого любые изменения структуры проекта, например, объема проекта или концептуального проекта, автоматически присваиваются этому пакету. Требуется лишь ввести соответствующий запрос на перенос.

 Создание организационной структуры проекта

        В SAP Solution Manager предусмотрена возможность для ввода данных об используемых в проекте организационных единицах. Информация, помимо названия подразделения, может включать функцию, страну, часовой пояс.      

Определение системной среды. Системные роли

        Для получения доступа к системам компонентов на последующих фазах проекта необходимо определить для шаблонного проекта или проекта внедрения системную среду.

        SAP Solution Manager позволяет централизованно управлять всеми логическими системами в рамках проекта. 

Данные системной среды проекта включают (см. рис 2.5):

  • Логический компонент;

  • Версию продукта;

  • Логические системы выбранных системных ролей.

   

Рисунок 2.5 – Определение системной среды.

        Логический компонент отражает набор логических систем (инстанций) для продукта. Выбор логического компонента влияет на выбор сценариев/процессов из Репозитария бизнес-процессов в фазе концептуального проекта. Выбрать можно только те сценарии/процессы, которые соответствуют определенному логическому компоненту и версии продукта.

        Вместе с SAP Solution Manager поставляются различные системные роли внедрения. Участники проекта могут входить в проект по внедрению с помощью присвоенных им ролей и полномочий и работать над определенными задачами проекта. Профили полномочий могут быть адаптированы к требованиям проекта. Для копирования и изменения ролей SAP Solution Manager используется Генератор профилей (PFCG).

        Концепция полномочий SAP Solution Manager предварительно определена для следующих ролей:

  • Руководитель проекта (SAP_SOLAR_PM);

  • Консультант по приложению (SAP_SOLAR_AC);

  • Консультант по разработке/базису (SAP_SOLAR_BC);

  • Технический консультант (SAP_SOLAR_TC).

        Каждая роль состоит из различных функций членов проектной команды, типичных для этой роли. Все вышесказанное иллюстрирует рисунок 2.6.

   

Рисунок 2.6 – Использование системных ролей SAP Solution Manager.

        В ходе проекта можно добавить системы для системных ролей. После сохранения информацию о логических системах для внешних продуктов (не SAP) изменить невозможно.

        Рассмотрим каждую роль и определенные для нее задачи проекта по внедрению.

        Руководитель проекта (SAP_SOLAR_PM)

Руководители проекта отвечают за управление и достижение необходимых результатов проекта. Они предупреждают отклонения от хода выполнения проекта и выполняют мероприятия по корректировке. Основные обязанности руководителя проекта (реализуются вSAP Solution Manager)  включают:

  • Разработка стратегий и процедур для зффективного внедрения SAP;

  • Поддержка определения и моделирования бизнес-процессов;

  • Конфигурация ПО в соответствии с бизнес-требованиями;

  • Определение глобальных и локальных требований стандартизации;

  • Определение требований для организационных изменений на предприятии;

  • Помощь в передаче знаний другим членам проектной группы;

  • Поддержка в организации и проведении тестов;

  • Определение требований для профиля полномочий и полномочий на доступ.

        Функции, определенные для роли руководителя проекта изображены на рисунке 2.7

   

Рисунок 2.7 – Функции для Руководителя проекта.

       Консультант по приложению (SAP_SOLAR_AC).

        Консультанты по приложениям отвечают за адаптацию концептуального бизнес-проекта и конфигурации ПО к бизнес-процессам, а также за выполнение требований по анализу и отчетности. Для выполнения этих задач применяются знания испытанных бизнес-процедур. Эта роль обладает ограниченными полномочиями и предназначена для участников проекта. Ее можно адаптировать к другим полномочиям.

Основные обязанности консультанта по приложению  включают:

  • Разработка стратегий и процедур для эффективного внедрения SАР;

  • Поддержка определения и моделирования бизнес-процессов;

  • Конфигурация ПО в соответствии с бизнес-требованиями;

  • Определение глобальных и локальных требований стандартизации;

  • Определение требований для организационных изменений на предприятии;

  • Помощь в передаче знаний другим членам проектной группы;

  • Поддержка в организации и проведении тестов;

  • Определение требований для профиля полномочий и полномочий на доступ.

        Функции, определенные для роли консультанта по приложению изображены на рисунке 2.8

   

Рисунок 2.8 – Функции для Консультанта по приложению.

        Консультант по разработке/базису (SAP_SOLAR_BC).

        Консультанты по разработке/Базису вместе с  руководителем проекта и консультантом по приложению работают над планированием и организацией концепции полномочий, Они также выполняют задачи по разработке и специфичные для клиентов разработки.

Основные обязанности консультанта по Базису включают следующие действия:

  • Организация полномочий;

  • Разработка АВАР-программ, например, интерфейсов, форм или модификаций;

  • Работа с инструментальными средствами АВАР;

  • Для дополнительных полномочий доступна роль SAP_BC_DWB_ABAPDEVELOPER, которая содержит полномочие на внедрение новых приложений и расширений, а также на изменение стандартных приложений с помощью инструментов м сервисов, предоставленных инструментальными средствами АВАР.

         Функции, определенные для роли консультанта по разработке/Базису изображены на рисунке 2.9

   

Рисунок 2.9 – Функции для Консультанта по Базису.

        Технический консультант (SAP_SOLAR_TC).

        Технические консультанты вместе с руководителями проекта и группы технических специалистов разрабатывают технические требования и затем выполняют необходимые технические задачи в системе. В зависимости от объема и сложности проекта внедрения, технические консультанты могут работать в нескольких областях, например системное администрирование, управление базой данных, управление сетями, управление операционной системой, разработка компонентов для всех приложений или АВАР-разработка.

        Основные обязанности технического консультанта включают следующие действия:

  • Настройка и установка системного ландшафта;

  • Управление системным ландшафтом и соответствующим ландшафтом переноса;

  • Обеспечение поддержки и помощи по всем техническим вопросам входе проекта внедрения ;

  • Разработка и реализация плана подготовки перед продуктивным запуском;

  • Эффективная коммуникация с группами по бизнес-процессам;

  • Эффективная передача знаний сотрудникам отдела системного администрирования.

        Функции, определенные для роли технического консультанта изображены на рисунке 2.10.

   

Рисунок 2.10 – Функции для Технического консультанта.

        Вернемся в системной среде. В зависимости от определяемой системной среды, будут доступны следующие функции:

  • Переход к к бизнес-содержимому в системах компонентов;

  • Создание IMG-проектов в системах компонентов и распределение центральных данных проекта;

  • Сравнение и распределение настроек в локальных системах.

        После выбора логических компонентов проекта и определения системы разработок в проекте SAP Solution Manager есть возможность увидеть все подпроекты настройки. Для подпроектов настройки можно создавать соответствующие IMG проекты и распределять центральные данные проекта (общие данные, информацию о сотрудниках, стандартах проекта, выбранных странах). После создания IMG проекта в системах компонентов вся соответствующая информация заносится в данные по объему проекта. Напротив каждого системного компонента есть поле, представляющее собой светофор. Зеленый и красный сигналы светофора указывают на то, было ли выполнено создание IMG проекта и распределение соответствующих данных или нет.
